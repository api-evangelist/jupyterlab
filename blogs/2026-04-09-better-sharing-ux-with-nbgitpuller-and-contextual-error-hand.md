<figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*d1wtukpXH7983Z0nCKv33Q.png" /></figure><h4>TL;DR</h4><p><a href="https://github.com/jupyterhub/nbgitpuller">nbgitpuller</a> now has improved UX context-aware error handling. Update to <a href="https://pypi.org/project/nbgitpuller/1.3.0/">version 1.3.0</a> and let us know what you think by opening an <a href="https://github.com/jupyterhub/nbgitpuller/issues">issue</a> 🚀</p><h4>What is nbgitpuller?</h4><p><a href="https://github.com/jupyterhub/nbgitpuller">nbgitpuller</a> lets you distribute content to a Jupyter user via the click of a button! This allows your users to focus on the content without needing to understand git or other version control machinery.</p><p>nbgitpuller provides <a href="https://nbgitpuller.readthedocs.io/en/latest/topic/automatic-merging.html">automatic, opinionated conflict resolution</a> by using git under the hood. It is commonly used to distribute content to multiple users of a JupyterHub, though it works just fine on an individual person’s computer, if they have Jupyter installed.</p><p>Here’s an example of nbgitpuller in action:</p><ol><li>The author uses the <a href="https://nbgitpuller.readthedocs.io/en/latest/link.html">nbgitpuller link generator</a> to create a link.</li></ol><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/0*2M4f_Hh6iyGP7VCo.png" /></figure><p>2. The consumer clicks the link, and the content is pulled into a live Jupyter session.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/0*J_A7MbZ2JI5hZh7n.gif" /></figure><p>Example use cases for sharing with nbgitpuller include:</p><ul><li>Interactive demos</li><li>Workshop and training scenarios</li><li>University exams</li></ul><p>and many more!</p><h4>How does nbgitpuller work?</h4><p>The `nbgitpuller` package is installed in the compute environment. The compute environment now has a `/git-pull` endpoint URL that can understand where to pull content from using URL parameters. Syncing content critically depends on `git` operations to `fetch`, `checkout`, `clone`, `merge`, `commit`, etc.</p><h4>When it goes wrong</h4><p>Based on the data sent through by the kind folks running the Berkeley DataHub, there were 2163 logs available, of which:</p><ul><li>983 (45%) were ‘merge’ conflicts</li><li>493 (23%) were ‘fetch’ errors</li><li>467 (22%) were ‘ls-remote’ errors</li></ul><figure><img alt="" src="https://cdn-images-1.medium.com/max/700/1*v1NGeWyH5Frl9QDJ2qed0Q.png" /><figcaption>Bar chart showing the most common errors users faced.]</figcaption></figure><p>The remaining errors (10%) were mostly ‘checkout’ errors. This represents the most <strong>frequent</strong> errors as seen by students.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/700/1*6sYjI1QPnc_yz6IiIw442g.png" /><figcaption>Bar chart showing the most unique errors link authors made.</figcaption></figure><p>In the same set of logs, there were 172 <strong>unique</strong> errors: most ‘ls-remote’ errors come from mistakes in the content repo URL. This represents the most uniquely common errors made by instructors.</p><h4>Merge conflicts</h4><p>If the link author changes content after the consumer clicks a link, then `nbgitpuller` needs to sync updates for the consumer on subsequent link clicks. The `nbgitpuller` merging strategy makes opinionated choices so that the link consumer <em>never</em> has to interact with git, and will always preserve the consumer’s working changes.</p><p>Things can go wrong when</p><ul><li>Consumers can diverge the git history if they perform a git commit</li><li>Authors can diverge the git history if they perform force push commits</li></ul><h4>Error UX (old)</h4><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*prhJ0nFlArAhfFwFFZbbLA.png" /><figcaption>Old UX for merge errors.</figcaption></figure><p>Problems with the old UX include:</p><ul><li>a scary terminal</li><li>difficulty for the user to figure out what went wrong</li><li>no suggestion for the user to fix the problem or a signpost to continue to the compute platform</li></ul><h4>Error UX (new)</h4><figure><img alt="" src="https://cdn-images-1.medium.com/max/998/1*iE8c5G51r3qL8ITBFiOZKg.png" /><figcaption>New UX for merge errors.</figcaption></figure><p>New improvements to the UX include:</p><ul><li>The terminal is closed by default, but you can optionally toggle it open</li><li>There is a copy to clipboard button to easily share the error log from the terminal</li><li>A more user-friendly and context-aware helper message is displayed</li><li>A link to the general documentation for reference</li><li>A ‘Proceed without syncing’ button takes the user to the compute platform without making any changes</li><li>In the case of merge errors, an extra <strong>Backup and resync</strong> button option is presented</li></ul><h4>Learn more</h4><ul><li><a href="https://nbgitpuller.readthedocs.io/en/latest/">nbgitpuller documentation</a></li></ul><p><strong>Acknowledgements</strong></p><ul><li><a

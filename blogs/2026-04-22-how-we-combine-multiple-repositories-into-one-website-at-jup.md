<p>The Jupyter Book project has content in a bunch of different repositories — <a href="https://github.com/jupyter-book/jupyter-book">user docs</a>, <a href="https://github.com/jupyter-book/blog">a blog</a>, <a href="https://github.com/jupyter-book/team-compass">a team compass</a>, and <a href="https://github.com/jupyter-book/jupyterbook.org">project &amp; community pages</a>. We recently finished an initiative to serve each set of content at jupyterbook.org, while keeping the source files separated in their repositories.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/789/1*5u-ku4TtFxBSmtmjTPqhfA@2x.png" /><figcaption>Jupyter Book builds tools and standards for communicating and publishing computational narratives.</figcaption></figure><p>Getting there required a combination of Netlify configuration and several improvements to MyST and the MyST theme along the way. This post walks through the pieces and how they fit together, so you can do something similar with a multi-repo project.</p><p><em>Cross-posted with this </em><a href="https://jupyterbook.org/blog/posts/2026/multi-repo"><em>Jupyter Book blog post</em></a><em> where you can see the MyST engine in action.</em></p><h3>Why is this useful?</h3><p>Two things bugged us about the old setup:</p><ol><li>We had multiple types of content embedded in one repository. In particular we had <strong>versioned user documentation</strong> alongside <strong>evergreen community documentation</strong> (e.g., <a href="https://jupyterbook.org/roadmap">the roadmaps page</a>). This meant we had duplicated content for some user documentation, and a slow lag time before it’d make it’s way into /stable.</li><li>Search engines treat subdomains as separate websites, so blog.jupyterbook.org wasn’t helping jupyterbook.org in search rankings.</li></ol><p>We wanted jupyterbook.org, jupyterbook.org/blog, jupyterbook.org/stable, jupyterbook.org/compass all under one roof, with each repo still deploying on its own via static websites.</p><p>We accomplished this with <strong>Netlify proxy rules</strong> and <strong>shared MyST configuration</strong>!</p><h3>Proxying sub-sites with Netlify rewrites</h3><p>We discovered that <a href="https://docs.netlify.com/routing/redirects/rewrites-proxies/">Netlify reverse proxy rules</a> are a great way to accomplish this. These essentially intercept queries to a URL path (eg. /blog/**) and serve content at that path from elsewhere (e.g., blog.jupyterbook.org). The browser sees jupyterbook.org/blog, but Netlify fetches the content from a totally different site behind the scenes.</p><p>We set up a <a href="https://github.com/jupyter-book/jupyterbook.org">“jupyterbook.org” repo</a> that serves the landing page and our evergreen community documentation, and proxies everything else:</p><ul><li>/ - served directly from the <a href="https://github.com/jupyter-book/jupyterbook.org">jupyterbook.org repo</a> (served on Netlify)</li><li>/blog - proxied from the <a href="https://github.com/jupyter-book/blog">blog repo</a> (served on Netlify)</li><li>/compass - proxied from the <a href="https://github.com/jupyter-book/team-compass">team-compass repo</a> (Served on Netlify)</li><li>/stable - proxied from the <a href="https://github.com/jupyter-book/jupyter-book">jupyter-book repo</a> (served on <a href="https://readthedocs.org">ReadTheDocs</a>)</li></ul><p>Here’s what the <a href="https://github.com/jupyter-book/jupyterbook.org/blob/main/netlify.toml">netlify.toml</a> looks like (simplified):</p><pre># Blog: proxied from another Netlify site<br />[[redirects]]<br />  from = &quot;/blog/*&quot;<br />  to = &quot;https://jupyter-book-blog.netlify.app/blog/:splat&quot;<br />  status = 200<br />  force = true<br /><br /># User docs: proxied from ReadTheDocs<br />[[redirects]]<br />  from = &quot;/stable/*&quot;<br />  to = &quot;https://jupyter-book.readthedocs.io/stable/:splat&quot;<br />  status = 200<br />  force = true</pre><p>Each repo deploys independently, meaning we can update the content at whatever cadence is best for that repository. The <a href="http://jupyterbook.org">jupyterbook.org</a> repo only needs updating when the nav structure changes. But making it all <em>feel</em> like one site took some work on the MyST side.</p><h3>Shared configuration with extends</h3><p>MyST’s <a href="https://mystmd.org/guide/external-references">extends: configuration</a> lets one project inherit configuration from another. We use this to share navbar, logo, and favicon config across all sub-sites from a single <a href="https://github.com/jupyter-book/jupyterbook.org/blob/main/docs/_site/site.yml">site.yml</a> in the core repo:</p><pre># In the blog's myst.yml<br />extends:<br />  - https://github.com/jupyter-book/jupyterbook.org/raw/refs/heads/main/docs/_site/site.yml</pre><p>If we change the configuration at that file, it’ll automatically be inherited in every other repository when we re-build it (usually done on each commit, or on a daily CRON job with GitHub Actions).</p><p>Here are a few useful things that we’ve shared across sites via the pattern above:</p><ul><li>Navigation bar URLs</li><li>Site footer configuration</li><li>Site branding (logo, titles, etc)</li><li>Announcement bars (when we need a temporary announcement to show up everywhere)</li><li>Internal domain configuration (see above)</li></ul><p>Along the way we <a href="https://github.com/jupyter-book/mystmd/issues/2126">added </a><a href="https://github.com/jupyter-book/mystmd/issues/2126">parts: support to the </a><a href="https://github.com/jupyter-book/mystmd/issues/2126">extends: key</a> to get this working.</p><h3>One gotcha: BASE_URL for asset paths</h3><p>MyST bakes BASE_URL into CSS, JS, and image paths at build time. If you build with BASE_URL=/ but serve at /blog, none of those load properly.</p><p>To fix this, each sub-site builds with BASE_URL set to its final path on <a href="http://jupyterbook.org">jupyterbook.org</a>. For example, here’s some <a href="https://github.com/jupyter-book/blog/blob/main/netlify.toml">Netlify config</a> from the blog that does this:</p><pre>[build]<br />  publish = &quot;publish&quot;<br />  # This ensures that `/blog` is appended to asset paths<br />  environment = { BASE_URL = &quot;/blog&quot; }<br />  command = &quot;&quot;&quot;<br />    cd docs &amp;&amp; myst build --html &amp;&amp; \<br />    mkdir -p ../publish/blog &amp;&amp; \<br />    # Here we move the HTML assets to the blog<br />    mv _build/html/* ../publish/blog/<br />  &quot;&quot;&quot;</pre><h3>internal_domains for cross-site navigation</h3><p>BASE_URL fixes asset loading, but it doesn’t fix navigation. Jupyter Book used to treat full URLs as “external” links, adding an extra icon and opening a new tab when clicked. This led to a bunch of tabs any time you navigated around <a href="http://jupyterbook.org">jupyterbook.org</a>, and was <a href="https://github.com/jupyter-book/mystmd/issues/2719">also reported by the Project Pythia team</a>.</p><p>We added an <a href="https://github.com/jupyter-book/myst-theme/pull/816">internal_domains option</a> to fix this:</p><pre>site:<br />  options:<br />    internal_domains: jupyterbook.org</pre><p>Now the theme treats any link to jupyterbook.org as internal. It provides same-tab navigation and no external link icon, even though the link is technically a full URL.</p><h3>Want to do this too?</h3><p>Hopefully this is a useful workflow for communities that have multiple repositories of content.</p><p>All of the config is in <a href="https://github.com/jupyter-book/jupyterbook.org">jupyter-book/jupyterbook.org</a>, and the tracking issue with the full story is <a href="https://github.com/jupyter-book/jupyter-book/issues/2528">jupyter-book/jupyter-book#2528</a>.</p><p>If you’ve got questions about this work, come chat with us at <a href="https://discord.mystmd.org">discord.mystmd.org</a>.</p><h3>Acknowledgements</h3><ul><li>Thanks to <a href="https://projectpythia.org/">Project Pythia</a> for providing additional guidance, collaboration, and support for this work.</li><li>Thanks to <a href="https://docs.earthscope.org/en/latest/intro.html">EarthScope</a> for giving user feedback about the usefulness of this as they explore converting their docs to the new MyST engine.</li></ul><img alt="" height="1" src="https://medium.com/_/stat?event=post.clientViewed&amp;referrerSource=full_rss&amp;postId=588c0197a369" width="1" /><hr /><p><a href="https://blog.jupyter.org/how-we-combine-multiple-repositories-into-one-website-at-jupyterbook-org-588c0197a369">How we combine multiple repositories into one website at jupyterbook.org</a> was originally published in <a href="https://blog.jupyter.org">Jupyter Blog</a> on Medium, where people are continuing the conversation by highlighting and responding to this story.</p>
